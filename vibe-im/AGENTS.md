# Vibe IM 协作指令

本文件作用于整个 `vibe-im` 工作区及其所有子目录，是本仓库最高优先级的协作纪律和架构约束来源。所有开发、排障、重构、测试、文档、迁移、脚本、服务启动和配置变更都必须遵守。

## 文档与语言

项目文档统一放在根目录 `docs/`：

- `docs/vibe-im.md`：产品、架构和使用说明。
- `docs/database.md`：数据库、迁移和数据访问说明。
- `docs/git使用.md`：Git 使用记录或操作备忘。
- `docs/草稿.md`：人工草稿，默认忽略，不主动编辑。

除 `AGENTS.md` 外，不得在 `app/` 或其他子目录继续沉淀项目文档。新增文档先判断归属并写入 `docs/` 对应文档；确需保留子目录 README 时，只能作为极短跳转说明，不能成为第二事实源。

仓库内新增或修改的项目文档默认使用中文。架构说明、设计记录、运行手册、排障记录、README、AGENTS 指令、开发计划和复盘总结都应使用中文；只有外部 API 字段、代码标识符、错误原文、命令输出或用户明确要求英文时才保留英文。已有英文文档后续修改时优先补充或改写为中文，避免扩大英文文档面。

## 产品边界

```text
Vibe IM = 纯即时通讯系统
PostgreSQL = 用户、会话、成员、消息、附件和好友关系事实源
本地 uploads = 文件正文事实源
聊天消息 = 用户可见沟通记录
```

系统当前只保留账号登录/退出/资料维护、管理员管理真人用户、好友、单聊、群聊、群成员管理、文字/图片/文件消息、WebSocket 实时收发、离线补拉、未读数、PostgreSQL 迁移、本地文件存储和基础健康检查。

## 基本纪律

- 沟通机制：当用户询问建议、方案、是否可行、如何优化或泛泛提到“进行编码”时，先以全面、客观、科学、基于本项目边界和类似行业成熟方案的立场回复；除非用户明确要求“直接改/实现/修复/落地/提交代码”等执行指令，否则不要直接编码。
- 稳定可靠：核心链路优先选择简单、可验证、可回滚的实现；登录、成员权限、消息序号、文件权限、迁移和健康检查必须可验证。
- 易用通用：默认界面面向普通聊天场景；核心模型只表达用户、会话、成员、消息、附件、好友等通用 IM 概念。
- 系统提示统一：错误、成功、警告等系统级提示必须走公共 `SystemNotice` 入口，居中展示；页面不得各自实现临时 toast、重复状态或重复样式。
- 业务无关：不新增业务专用核心表、启动行为或硬编码垂直业务逻辑；新业务通过独立方案重新设计，不塞入当前纯 IM 主链路。
- 商用可延展：默认能力、部署方式、命名和数据模型应能面向不同行业客户解释和交付。
- 领域归属：新增能力前先判断属于 `chat/group`、`file`、`auth`、`admin`、`db` 还是 `realtime` 层。
- 复用优先：相同代码、函数或概念只能有一个主要实现入口，禁止在不同页面、API 或脚本中复制校验、状态变更、消息文案、文件写入或数据库副作用。
- 清理优先：每次重构或功能完善都同步识别并删除无用、重复、临时、废弃或误导代码；不长期保留无入口代码、重复 UI、过期脚本、死分支或失效文档。
- 小步验证：优先做小步、可回滚改动，并补充针对性验证。

## 上下文与防失忆纪律

不要把聊天上下文当作长期事实源。上下文压缩、线程恢复或长时间开发后，必须先回到项目事实源，而不是仅凭压缩摘要继续判断。

事实源优先级：

1. `AGENTS.md`：最高优先级协作纪律、架构边界和禁止事项。
2. `docs/*.md`：产品、架构、数据库、设计决策和运行说明。
3. 代码、迁移、测试和脚本：真实实现与可验证行为。
4. `git diff` / `git status`：当前未提交变更和工作区状态。
5. 对话摘要：只能作为线索，不能作为最终事实源。

压缩后或恢复工作前必须执行的恢复动作：

- 重新阅读 `AGENTS.md`。
- 阅读当前任务相关的 `docs/` 文档。
- 查看 `git status --short` 和相关 `git diff`。
- 重新定位最近修改过的关键文件和统一入口。
- 继续编码前确认当前需求、已完成内容、未完成内容和验证状态。

长期规则、架构边界、接口约定、校验规则、状态机、错误处理和系统提示规范，必须沉淀到 `AGENTS.md`、`docs/`、代码或测试中，不能只留在聊天记录里。

如果发现压缩摘要、历史对话、文档和代码互相冲突，以当前仓库事实为准；无法判断时，先说明冲突并提出最小确认点，不得基于记忆直接改核心链路。

## 聊天与群聊

- 生产代码发送消息必须走 `chat-service`，不能从无关层直接写 `messages` 或手动广播 WebSocket。
- 低层 `chat` repository 只负责消息序号、落库、成员表读写等数据库事实。
- 完整发送能力必须包含权限、成员关系、落库、会话更新时间、事件发布和实时投递。
- `realtime` 只负责 WebSocket 连接、请求响应和 chat event 分发，不能拥有业务数据库副作用。
- 群聊提示、成员加入/邀请/移除/解散、消息序号、系统通知、文件上传和权限校验必须由对应基础服务统一提供。
- 聊天历史消息只展示发送者身份快照，不绑定动态状态。
- 单聊不要求好友关系，但必须校验目标用户有效性并禁止与自己发起单聊；群聊邀请必须校验目标用户有效性、成员上限和重复成员，不要求双方是好友，不要求群主或管理员权限。

## 文件与数据库

- 图片和文件上传必须经过统一 API。
- 附件下载必须校验会话成员权限。
- 文件正文写入 `uploads/`，数据库只保存元数据和存储路径；不允许在数据库保存大文件正文。
- PostgreSQL 是应用唯一数据库。
- 数据库变更必须通过 `app/db/migrations/*.sql` 版本化迁移，禁止在运行时代码中直接新增表或索引。
- 调试运行时问题前，必须确认环境、数据库连接和实际加载的 `.env.local`。

## 系统不可用或数据异常时

先稳定和诊断：

1. 如果服务可能继续修改状态，先停止或隔离运行中的服务。
2. 确认端口由哪个进程占用。
3. 确认服务连接的是哪个数据库。
4. 检查用户、会话、成员、消息和附件状态。
5. 识别被违反的边界或缺失的基础设施。
6. 编辑前先提出最小修复计划。

除非用户明确批准数据修复，否则不要手动修改业务数据来让单个场景看起来已经修好。
