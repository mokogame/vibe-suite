做到“接近行业顶尖的 Agent Runtime”，把 `Agent 职责定义、上下文编排、记忆检索、工具权限、评估回归、可观测性` 做成一套系统。

当前 `vibe-claw` 已经有基础闭环，下一步应按 Runtime 核心能力升级。

**目标定义**
做到这几个标准，才算成熟：
- Agent 有结构化职责契约，而不是只有一段 instruction。
- 每次模型调用前都有可解释的上下文构建流程。
- 记忆不是全量塞进去，而是按相关性、重要性、时间、可信度检索。
- 历史消息不会无限堆叠，而是短期窗口 + 滚动摘要 + 长期记忆组合。
- 每次回答可审计：用了哪些记忆、哪些历史、哪些工具、为什么裁剪。
- 有评估集，能证明改动没有让 Agent 质量倒退。
- 对外 API 稳定，支持 SaaS 多租户、权限、租约、用量、限流。

**实施总路线**
我建议分 6 个阶段推进。

**阶段 1：结构化 Agent 职责契约**
当前 Agent 主要靠 `instruction`。要升级为 `Agent Contract`。

新增字段：
- `role`：角色，例如“小说写作教练”“客服机器人”“数据分析助手”。
- `mission`：核心目标。
- `boundaries`：不能做什么。
- `style`：语气、格式、详细程度。
- `outputContract`：输出格式要求。
- `toolPolicy`：允许调用哪些工具，什么条件下调用。
- `memoryPolicy`：哪些内容可写入长期记忆。
- `handoffPolicy`：什么时候交给其他 Agent 或人工。
- `safetyPolicy`：敏感信息、权限、拒答边界。
- `version`：Agent 配置版本。

落地文件：
- `vibe-claw/src/types.ts`
- `vibe-claw/src/store/store.ts`
- `vibe-claw/src/store/postgres-store.ts`
- `vibe-claw/src/api/routes/agent-routes.ts`
- `vibe-claw/src/api/admin-page.ts`

结果：
- Agent 不再只是“一个 prompt”，而是一个可治理的服务单元。

**阶段 2：重构 Context Builder**
当前上下文是在 route 里拼装，规则偏简单。应抽成独立模块。

新增：
- `src/core/context-builder.ts`
- `src/core/prompt-compiler.ts`

上下文分层：
- `system`：平台规则、安全边界。
- `developer`：Agent Contract、职责、输出要求。
- `memory`：长期记忆检索结果。
- `history`：最近对话窗口。
- `summary`：滚动摘要。
- `tool`：工具结果。
- `attachment`：附件内容或元数据。
- `user`：当前用户消息。

每个 Context Block 应包含：
- `source`
- `role`
- `content`
- `priority`
- `tokenCost`
- `recency`
- `relevance`
- `importance`
- `sensitive`
- `provenance`
- `reason`

结果：
- 每次模型调用都能解释“为什么这些内容进入上下文，为什么那些被裁掉”。

**阶段 3：升级记忆系统**
当前记忆是手动写入 + active 全量注入。顶尖方案必须支持“检索、排序、治理”。

新增能力：
- 语义检索：embedding + pgvector。
- 混合检索：语义相似度 + 关键词 + 最近访问 + 重要性。
- MMR 去冗余：避免相似记忆重复塞入。
- 记忆重要性评分：`importance`。
- 记忆可信度：`confidence`。
- 记忆来源：`provenance`。
- 记忆过期时间：`expiresAt`。
- 记忆最近访问：`lastAccessedAt`。
- 记忆命中审计：`memory_retrieval_logs`。
- 自动记忆候选：从对话中抽取“可能值得保存”的内容。
- 人工确认模式：重要记忆可先进入 `pending`，管理员确认后 active。

推荐表结构扩展：
- `agent_memories.importance`
- `agent_memories.confidence`
- `agent_memories.tags`
- `agent_memories.embedding`
- `agent_memories.expires_at`
- `agent_memories.last_accessed_at`
- `agent_memories.provenance`
- `agent_memory_events`

结果：
- 记忆从“资料库”升级为“可治理的长期记忆系统”。

**阶段 4：成熟对话上下文策略**
每次聊天应该采用组合策略：

1. 当前消息永远保留。
2. 最近 N 条消息保留。
3. 更早历史进入 rolling summary。
4. 长期记忆按相关性召回。
5. Agent Contract 永远保留。
6. 工具结果按任务需要注入。
7. 超预算时优先裁掉低相关、低重要、旧的上下文。
8. 每次压缩生成审计记录。

压缩策略升级：
- `none`
- `recent_only`
- `rolling_summary`
- `semantic_recall`
- `hybrid`
- `protocol_minimal`
- `contract_strict`

当前 `rolling_summary` 是假摘要，后续应改为：
- 用模型生成稳定摘要。
- 摘要分为事实、用户偏好、未完成任务、决策记录。
- 摘要可版本化和回滚。

结果：
- 长对话不会越来越乱，也不会因为历史太长丢关键事实。

**阶段 5：工具调用与权限治理**
当前工具调用是显式传入，不支持模型自主选择工具。成熟方案需要“可控的工具调用”。

不是让模型随便执行工具，而是：
- 工具注册表声明 schema。
- Agent Contract 声明允许工具。
- 模型只可请求工具调用。
- Runtime 校验权限、租约、scope、风险等级。
- 高风险工具需要确认或审批。
- 工具结果进入上下文。
- 所有工具调用写审计。

新增：
- `tool_call_requested`
- `tool_call_approved`
- `tool_call_executed`
- `tool_call_failed`

结果：
- Agent 可以真正为外部服务执行任务，但不会越权。

**阶段 6：评估与质量门禁**
没有评估，就不能证明“顶尖”。

新增：
- `eval_cases`
- `eval_runs`
- `eval_scores`
- `golden conversations`

评估维度：
- 是否遵守职责。
- 是否正确使用记忆。
- 是否不泄露不该泄露的上下文。
- 是否输出格式稳定。
- 是否能处理长对话。
- 是否能处理冲突记忆。
- 是否能处理工具失败。
- token 成本是否可控。
- 延迟是否可控。

CI 增加：
- 单元测试：context builder、memory retrieval、prompt compiler。
- 集成测试：真实/Mock provider。
- 回归测试：固定 Agent + 固定对话集。
- 安全测试：prompt injection、越权 memory、跨 tenant 泄露。

结果：
- 每次升级都有质量基线，不靠主观感觉。

**优先级建议**
我建议这样推进：

1. `P0`：Agent Contract 数据结构和后台配置。
2. `P0`：Context Builder 独立模块。
3. `P0`：Provider messages 分层，不再全部塞 system。
4. `P1`：Memory retrieval，先做关键词 + priority + recency。
5. `P1`：pgvector embedding 检索。
6. `P1`：滚动摘要真实生成。
7. `P2`：自动记忆候选提取。
8. `P2`：工具调用请求/执行闭环。
9. `P2`：评估集和回归测试。
10. `P3`：多 Agent handoff、评审 Agent、仲裁 Agent。

**第一轮具体交付**
如果我们开始做，我建议第一轮只做 Runtime 核心，不碰太多 UI：

- 新增 `context-builder.ts`
- 新增 `prompt-compiler.ts`
- 新增 Agent Contract 类型
- 修改 Agent 创建/更新 API
- 修改 Provider 调用 messages 结构
- 修改 message flow 使用 Context Builder
- 增加 context audit 详情
- 增加测试覆盖：
  - Agent instruction 永远保留
  - 当前用户消息永远保留
  - 最近历史优先保留
  - memory 只注入 active 且同 scope
  - 超 token 后低优先级内容被裁剪
  - sensitive 内容脱敏
  - conversation lock 不回归

**判断**
按 Runtime 平台来建设，不要只在后台 UI 上修补。
先做：`Agent Contract + Context Builder + Prompt Compiler`。这是整个系统从“能用”升级到“专业 Agent 平台”的地基。
