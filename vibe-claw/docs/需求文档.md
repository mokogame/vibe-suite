# Vibe Claw 需求

## 目标定位

Vibe Claw 的目标从内部联调 MVP 升级为正式对外 SaaS/API 服务。

系统必须支持第三方通过 API 安全、稳定、可观测地调用智能体；支持多租户、持久化、异步任务、会话、记忆、模型 Provider、Webhook 和后台运营管理。

当前状态可作为内部联调和低并发试用基础，但距离正式 SaaS/API 上线还需要补齐生产级能力。

## 当前已具备能力

- Bearer Token 鉴权。
- API Token 和 scope 权限。
- Agent 创建、修改、对话、记忆管理。
- Provider 配置，支持 DeepSeek / OpenAI-compatible。
- 异步 Run 创建、查询、取消。
- Run 队列、事件、审计。
- Webhook 回调基础能力。
- PostgreSQL 持久化。
- 会话历史、长期记忆、上下文压缩。
- 管理后台和 OpenAPI 基础入口。

## 当前并发能力确认

- 支持多个外部请求同时创建 Run。
- 支持多个 Run 并发执行，默认并发由 `VIBE_CLAW_RUN_CONCURRENCY` 控制。
- 每个 Run、Step、Message、Event、Audit 均有独立 ID，PostgreSQL 持久化下不同 Run 不应混数据。
- 一个 Run 内传入多个 `agentIds` 时，当前是顺序协作，不是同一 Run 内并行协作。
- 同一个 conversation 同时收到多条消息时，目前没有会话级锁，存在上下文读取竞态。
- 当前队列是进程内队列，虽然任务状态落库，但多实例部署时仍缺少分布式 claim/lease，存在重复执行风险。

## P0：正式 SaaS/API 上线前必须完成

### 1. 多租户隔离

所有核心资源必须增加 `tenant_id` / `project_id`，并在 API 层强制过滤。

覆盖范围：

- Provider
- Agent
- Conversation
- Message
- Memory
- Run
- Run Step
- Run Event
- Artifact
- API Token
- Lease
- Audit Event
- Webhook

验收要求：

- 任意 API Token 只能访问所属 tenant/project 的数据。
- 管理后台只展示当前 tenant/project 数据。
- 审计日志记录 tenant/project、actor、source IP 和 user-agent。

### 2. Idempotency-Key

外部写接口必须支持幂等键，避免调用方网络重试导致重复创建资源。

覆盖接口：

- `POST /v1/runs`
- `POST /v1/agents/:id/messages`
- `POST /v1/agents`
- `POST /v1/providers`
- `POST /v1/tokens`
- `POST /v1/agents/:id/memories`
- `POST /v1/agents/:id/leases`

验收要求：

- 相同 tenant、actor、method、path、Idempotency-Key、body hash 返回同一结果。
- body hash 不一致时返回冲突错误。
- 幂等记录有 TTL 和清理机制。

### 3. Conversation 串行锁

同一个 conversation 的消息写入和模型调用必须串行处理。

验收要求：

- 同一 conversation 并发发送多条消息时，系统按提交顺序处理。
- 后一条消息必须读取前一条完成后的历史上下文。
- 锁超时后必须可恢复，不允许永久阻塞。

### 4. 分布式队列

当前进程内队列必须升级为可多实例部署的队列。

可选实现：

- PostgreSQL claim/lease 队列。
- Redis/BullMQ。

验收要求：

- 多个服务实例同时运行时，同一任务只会被一个实例执行。
- worker 崩溃后任务可超时回收并重试。
- 支持最大重试次数、退避策略、死信状态。
- `/v1/queue` 能展示 queued/running/completed/failed/dead-letter 统计。

### 5. 限流、配额和成本预算

必须按 token、tenant、agent、provider 维度限制调用。

验收要求：

- 支持 QPS 限流。
- 支持并发数限制。
- 支持每日/每月 token 额度。
- 支持成本预算和超额拒绝。
- 超限返回明确错误码和剩余额度信息。

### 6. Secret 管理

生产环境不得依赖本地明文文件或直接暴露密钥值。

验收要求：

- Provider 只保存 secret 引用，不保存明文 key。
- 运行时从 Vault/KMS/云 Secret Manager 或等价机制读取。
- 后台展示必须脱敏。
- 审计日志不得记录明文 key。

### 7. Webhook 可靠投递

Webhook 必须具备生产级可靠性。

验收要求：

- 支持签名。
- 支持重试和指数退避。
- 支持投递日志查询。
- 支持死信状态。
- 支持手动重放。

### 8. 流式响应

对话接口必须支持流式输出。

验收要求：

- 支持 SSE 或 WebSocket。
- 输出 token/片段可实时返回给调用方。
- 流结束后完整消息落库。
- 流失败时保留错误事件和可查询状态。

### 9. 可观测性

正式服务必须具备基础 observability。

验收要求：

- 结构化日志。
- requestId / runId / tenantId 全链路贯穿。
- Provider latency、token usage、error rate 指标。
- 队列积压、失败率、Webhook 投递失败告警。
- 支持 trace 或等价链路追踪。

### 10. 数据库迁移治理

迁移必须适合生产环境。

验收要求：

- 所有新增表和字段通过增量 migration 管理。
- 迁移可重复执行且幂等。
- 迁移前后有校验命令。
- 明确不可回滚迁移的风险说明。

## P1：生产增强能力

### 1. Agent 版本管理

- 支持 draft / published / archived 状态。
- 支持版本发布、回滚、灰度。
- Run 必须记录使用的 Agent 版本。

### 2. Protocol Run 完整闭环

当前 Protocol Run 需要升级为真实模型调用闭环。

验收要求：

- 模型按协议生成 JSON。
- 输出按 JSON Schema 校验。
- 校验失败时自动修复或重试。
- 最终返回结构化结果、原始文本、校验错误和 usage。

### 3. Memory 生命周期

- 支持记忆去重。
- 支持记忆合并。
- 支持过期策略。
- 支持人工审核和拒绝。
- 支持按 scope 检索和注入。

### 4. 多 Provider 路由

- 支持主备 Provider。
- 支持失败降级。
- 支持按成本、延迟、可用性选择模型。
- 支持 Provider 熔断。

### 5. 后台权限模型

- 管理后台不应只依赖单个 Bearer Token。
- 支持用户、角色、权限。
- 支持租户管理员和只读审计角色。

### 6. 评测与质量管理

- 支持 Agent 测试集。
- 支持回归评测。
- 支持输出评分和人工标注。
- 支持发布前质量门禁。

## 既有待办保留

### 提交前代码审查

检查大改是否存在临时实现、重复逻辑、命名不稳、过宽类型、测试只覆盖 happy path 的问题。

### 整理迁移

如果项目已有人跑过迁移，所有新增结构必须拆成增量迁移，不能继续追加到初始迁移里。

### 补 OpenAPI 完整 schema

OpenAPI 必须覆盖普通对话、协议、记忆、租约、后台入口、Webhook、幂等、限流错误等 request/response schema。

### Provider 配置真正接入运行时

Provider 配置必须支持 Agent 或 Run 选择，并作为运行时调用路径的唯一来源。环境变量 Provider 仅作为默认 fallback。

### 后台 UI 从入口页变成可操作页面

后台必须覆盖模型配置、Agent、会话、调用记录、记忆、Token/租约、审计、队列、Webhook、租户和用量页面。

### 持久化队列

当前队列需要升级为生产可用的 DB 队列或 BullMQ，并支持多实例安全执行。

## 最小正式上线门槛

正式对外 SaaS/API 上线前，至少必须完成：

- 多租户隔离。
- Idempotency-Key。
- Conversation 串行锁。
- 分布式队列 claim/lease。
- 限流、配额和成本预算。
- Secret 管理。
- Webhook 重试和死信。
- 基础监控、日志和告警。

未完成以上任一项，不应作为正式 SaaS/API 服务对外开放。
